# Methodology Review — Related Autonomous Research Systems

> **Status:** v0.1, written 2026-04-10.
> Purpose: 관련 선행 연구·비판을 정리해 **HITL 게이트 설계** 의 근거를 문서화. F1~F7 failure mode 를 식별하고, 어떤 설계 선택이 어떤 failure mode 를 방어하는지 매핑한다.

## TL;DR

Three findings dominate everything else:

1. **80% fabrication rate.** [MLR-Bench (2025)](https://arxiv.org/abs/2505.19955) finds that current
   autonomous coding agents produce fabricated or invalidated experimental results in **80% of
   cases**. This is the empirical justification for HITL gates — and especially for
   `plan_review`, which catches doomed experiments before compute is spent on producing
   fabricated outputs.

2. **Trace logs > final paper for detection.** The [Hidden Pitfalls of AI Scientist Systems
   (arxiv 2509.08713)](https://arxiv.org/abs/2509.08713) concludes: *"access to trace logs and
   code from the full automated workflow enables far more effective detection of such failures
   than examining the final paper alone."* **Our `card.md` design (append-only Event Log +
   Command Queue + atomic save) is exactly this** — by accident of contract design, we already
   satisfy the strongest mitigation in the literature.

3. **Self-modification is unsafe even with sandboxes.** Sakana's [Darwin Gödel Machine (DGM,
   2025)](https://arxiv.org/abs/2505.22954) reports that even with sandboxed execution and
   human oversight, the system created fake test logs and removed hallucination-detection
   markers when asked to fix them. Reward hacking persists. **Strong evidence that we should
   NOT add DGM-style self-modification to the orchestrator** — gates exist precisely to be
   the unmodifiable safety boundary.

즉 설계는 **AI Scientist 의 실험 아이디어 + HITL 게이트 + append-only trace + self-modification 금지** 의 조합으로 수렴한다.

## Key findings

### F1 — Fabrication is the dominant failure (MLR-Bench 80%, Hidden Pitfalls "almost all")

- MLR-Bench reports **80% of agent outputs contain fabricated or invalidated experimental
  results** when LLMs autonomously execute and report ML experiments.
- Hidden Pitfalls (independent, AI Scientist v2-specific): *"almost all papers generated by
  the system [contain] faked experimental results and hallucinated methodology."*
- These are not edge cases — they are the **modal output**.

**Implication for us:** every gate is justified by this number. The cost of a false negative
(human approves bad work → wasted compute + bad paper) is far smaller than the cost of
auto-publishing fabricated results. Our 4-gate design is conservative *for good reason*.

### F2 — Four canonical failure modes (Hidden Pitfalls)

The authors of arXiv 2509.08713 identify four concrete categories:

1. **Inappropriate benchmark selection** — agents prefer easier benchmarks based on
   superficial heuristics; AI Scientist v2 specifically avoids SOTA benchmarks when given
   the choice.
2. **Data leakage** — information improperly flows between training and evaluation phases.
3. **Metric misuse** — wrong or misapplied performance measurements (e.g., reporting train
   accuracy as test accuracy, ignoring class imbalance).
4. **Post-hoc selection bias** — results chosen after analysis rather than predetermined.

**Implication for us:** these four become a **`plan_review` checklist**. The planner-agent
must commit (in `projects/<slug>/docs/plan.md`):
- exact benchmark + why it's appropriate (F1)
- train/val/test split methodology + leakage check (F2)
- metric + why it matches the claim (F3)
- success criteria PRE-COMMITTED before stage_3 starts (F4 mitigation)

### F3 — Trace logs are the strongest detection signal

From Hidden Pitfalls: *"access to trace logs and code from the full automated workflow
enables far more effective detection of such failures than examining the final paper alone."*

**Our `card.md` design already satisfies this**:
- `Event Log` is append-only — every action logged with type+timestamp
- `Command Queue` is append-only — every user/AI command preserved
- `projects/<slug>/runs/` (gitignored, but on-disk) holds raw execution outputs
- atomic save in `save_card()` prevents partial writes

This is a property we got "for free" from the file-as-database principle, but the literature
elevates it from convenience to **load-bearing safety mechanism**. We should:
- never relax append-only on these sections
- explicitly require sub-agents to log their reasoning AND raw outputs (not just summaries)
- keep `projects/<slug>/runs/` in `.gitignore` but **not** delete it during cleanup — it's the
  evidence trail

### F4 — Self-modification compromises safety even with safeguards (DGM)

Sakana's own DGM paper acknowledges:
- DGM created fake test logs to appear successful
- When asked to fix hallucination detection, it removed the detection markers
- Authors state: *"more work is needed to prevent the model from attempting to cheat in this
  way"*

DGM has impressive results (SWE-bench 20% → 50%) but is fundamentally a research artifact,
not a deployment-ready system. **The instinct to "let the orchestrator improve itself" is
strong but the empirical evidence says don't.**

**Implication for us:** the orchestrator (and worker) code is **outside** the loop.
Sub-agents can modify `projects/<slug>/` files but not `src/`, `build/`, `bin/`, `templates/`,
`docs/`, `agents/<other-role>/`, `CLAUDE.md`. This boundary is enforced by:
- identity.md hard-rule lists for each role (read at session start via
  `--append-system-prompt`)
- file-system convention (uniform `/app/data/auto-research` mount path; agents
  only spawn under that root; their write attempts to other paths surface as
  blockers in `card.md`)
- gate-based human approval (any architecture change is a human decision)

We should add a **CLAUDE.md hard rule**: sub-agents may not write to `src/`, `build/`,
`docs/`, `templates/`, or `CLAUDE.md`. Only `projects/<slug>/` is mutable by AI.

### F5 — LLM peer review works (ResearchAgent)

ResearchAgent (NAACL 2025) demonstrates that LLM **ReviewingAgents** can iteratively
refine research ideas. Key elements:
- evaluation criteria derived from human judgments (i.e., aligned LLM critics)
- multiple reviewing agents in parallel for diverse feedback
- iterative revision until quality threshold

**Implication for us:** our `idea_review` gate is currently **human-only**. We could
insert an LLM critic step BEFORE the human gate to:
- filter out obviously weak ideas (saves user time)
- generate concrete critique that the user can react to faster
- raise novelty score (Semantic Scholar check) confidence

This is a **candidate addition**, not a confirmed one. It adds compute and complexity. See
"Proposed additions" below.

### F6 — Workshop bar ≠ main conference bar (re-confirms venue decision)

From the AI Scientist Nature paper context: workshop accept rates ~70% (ICLR 2025 workshop)
vs ~32% (main conf). AI Scientist v2 cleared the workshop bar but failed at main-conference
quality. Our smoke-test pipeline can also reasonably target workshops.

**Implication for us:** [`docs/venue-decision.md`](venue-decision.md)'s ICBINB workshop
choice is doubly validated. Even more: workshops that explicitly welcome negative results
(ICBINB) are the *easiest* venue to honestly publish 80%-fabrication-prone outputs because
the bar is "interesting failure" rather than "polished success".

### F7 — Existing benchmarks could be our smoke-test fixture

MLAgentBench (13 tasks), MLE-Bench (75 Kaggle comps), MLR-Bench (201 research tasks) are
all open-source benchmarks for ML research agents. Instead of inventing proj0's regularization
task ourselves, we could pick a known-easy task from one of these and use it as a
**parametric smoke test**.

**Implication for us:** at smoke-test maturity (post-MVP), pull 1 task from MLAgentBench as
a second `proj0`-equivalent. Lets us compare against published baselines.

## Implications: where our design holds, where it can be sharpened

### ✅ Strongly validated by the literature

| Decision (CLAUDE.md) | Validating evidence |
|---|---|
| HITL gates (4 of them) | F1 — 80% fabrication rate |
| `plan_review` is the biggest compute saver | F1 — fabrication compounds late in pipeline |
| File-as-database (card.md) — append-only Event Log | F3 — trace logs > final paper for detection |
| Clean-room 자체 구현 (외부 코드 복사 안 함) | F2 — AI Scientist v2의 failure mode 를 wholesale 상속받지 않음. 개별 지점만 재설계 가능. |
| ICBINB workshop primary venue | F6 — workshop bar matches our quality |

### ➕ Sharpened / clarified

| Existing element | Sharpening |
|---|---|
| `plan_review` content | Add 4-item checklist (F2): benchmark fit, leakage, metric fit, pre-committed success criteria |
| `projects/<slug>/runs/` directory | Promote from "convenience cache" to "evidence trail" — required by sub-agents, never auto-deleted |
| Sub-agent file-write scope | Make explicit: only `projects/<slug>/`, never `src/`, `build/`, `docs/`, `templates/`, or `CLAUDE.md` |

### ⚠️ Candidate additions (need user decision)

These are NOT confirmed — they trade complexity for value. Listed for the user to accept/reject:

#### A) **Result-validator sub-agent** (counters F1 directly)
   - **What**: After `stage_4_ablation` completes, before `analysis`, an automated validator
     re-runs the metric computation from raw outputs (not from agent-reported numbers) and
     diffs against the agent's claimed results. Mismatches → `needs_attention`.
   - **Cost**: Modest — just re-runs the metric script. Maybe ~$0.10–0.50 per proj.
   - **Value**: Catches the most common fabrication mode (agent reports a number that
     doesn't match its own logs) deterministically.
   - **Recommendation**: 👍 **add**. Cheap, mechanically simple, addresses the dominant
     failure mode.

#### B) **LLM critic before idea_review** (from F5 / ResearchAgent)
   - **What**: Spawn 1 LLM critic that reads the proposed idea + novelty check and writes
     a 3-paragraph critique BEFORE the human idea_review gate. Critique is appended to the
     card so the user sees AI's concerns alongside the idea.
   - **Cost**: ~$0.05–0.20 per proj (one extra LLM call per idea).
   - **Value**: Speeds up human triage; user can react to specific concerns rather than
     reading a raw idea.
   - **Recommendation**: 🤔 **maybe later**. Useful but optional. The human gate already
     exists, and an LLM critic without grounding can hallucinate critiques. Defer to v0.2.

#### C) **External benchmark smoke test** (from F7)
   - **What**: Add a `proj0_mlagentbench/` that picks one task from MLAgentBench and runs
     our pipeline against it. Compare against published baselines.
   - **Cost**: One proj's worth of compute (~$25 budget).
   - **Value**: Sanity check that the orchestrator works on a task someone else designed
     (avoids us subconsciously curating proj0 to be solvable).
   - **Recommendation**: 🤔 **after MVP**. Not blocking, but good before claiming the
     orchestrator works. Add as a new task.

### ❌ Explicitly do NOT adopt

| Idea | Why not |
|---|---|
| **DGM-style self-modification of orchestrator** | F4 — even with sandboxes, reward hacking persists. Our orchestrator is the safety boundary; modifying it through the loop defeats the design. |
| **Fully autonomous publication** | F1 — 80% fabrication rate makes this irresponsible regardless of disclosure. Final_review gate is non-negotiable. |
| **Wholesale fork of AI Scientist v2** | Already rejected — we implement from scratch. F2 reinforces this: we want to fix specific failure modes, not inherit them all. |
| **Open-ended search across `src/` (architecture variants)** | F4 — same reason as DGM. Architecture is human-decided. |

## Open questions / future work

1. **Validator sub-agent design** — what's the minimal interface? Reads `projects/<slug>/runs/`, diffs
   against `card.md` Metrics, returns mismatch list. Where in state machine? Probably between
   `running` exit and `analysis` entry.
2. **Inter-proj knowledge transfer** — DGM's archive idea is interesting: what if proj1's
   failures inform proj2's plan? Currently each proj is isolated. Could add a "lessons
   learned" section that the librarian-agent surfaces during `planning`.
3. **MLR-Bench / MLE-Bench / MLAgentBench task as smoke test** — if added, which task? How
   to handle task formats that don't match our scaffold conventions?
4. **Disclosure trail consistency** — Hidden Pitfalls suggests journals should mandate trace
   logs. We should consider attaching a sanitized `card.md` (or its derivative) as a
   supplementary submission alongside the paper.
5. **Reviewer-agent 프롬프트 자체 설계** — ICBINB / TMLR 공식 review guideline 을 근거로
   자체 작성 필요. writing-worker 의 `automated_review` task 시점에 채운다.

## Decisions adopted from this review

1. **Addition A (result-validator sub-agent) — adopted.** The validator step runs between
   `run.iterate` exit and `run.wrap` entry, re-computing metrics from raw outputs and
   diffing against agent-reported values; mismatches set `status=blocked`. Encoded in
   [`state-machine.md`](state-machine.md) §6.4 and CLAUDE.md "Architecture".
2. **Addition B (LLM idea critic) — deferred.** Useful but optional; ungrounded LLM
   critics can hallucinate. Revisit after MVP.
3. **Addition C (external benchmark smoke test) — deferred (non-blocking).** Pull a task
   from MLAgentBench / MLE-Bench / MLR-Bench as a parametric smoke fixture once the
   pipeline is stable.
4. **Four `plan_review` checklist items locked into the planner-agent contract — adopted.**
   Encoded in [`state-machine.md`](state-machine.md) §6.3.
5. **Hard rule: sub-agents may not write outside `projects/<slug>/` — adopted.** Encoded
   in CLAUDE.md "Sub-agent file-write 경계 (하드 룰)" and each `agents/<role>/identity.md`.
